# managed_gc – Управляемый рантайм с мультипоточной аллокацией и сборщиком мусора

**managed_gc** — это экспериментальная реализация управляемой кучи, многопоточного lock-free аллокатора и сборщика мусора (Mark & Sweep) на языке Rust.  
Проект вдохновлён архитектурой аллокатора Go: спаны памяти, локальные кэши потоков и центральный пул. Он предназначен для исследовательских целей и демонстрации возможностей языка при построении высокопроизводительных управляемых сред.

## 🚀 Ключевые возможности

- **Мультипоточная аллокация без глобальных блокировок** – каждый поток использует локальный кэш (`MCache`) и атомарные операции, что минимизирует конкуренцию.
- **Собственная система классов размеров** – от 64 байт до 3072 байт, с автоматическим подбором спанов нужной ёмкости.
- **Сборщик мусора (Mark & Sweep)** – корректно обрабатывает циклические ссылки и освобождает недостижимые объекты.
- **Переиспользование памяти после GC** – освобождённые спаны возвращаются в центральный пул (`MCentral`) через `sweep_full_spans` и доступны для повторной аллокации без обращения к ОС.
- **Write barrier (инфраструктура)** – заложена инфраструктура для инкрементального GC: `write_reference`, `barrier_worklist`, флаг `is_marking`.
- **Кросс-платформенное профилирование** – сравнение с ptmalloc2, libmalloc, jemalloc и mimalloc на Apple Silicon и AMD Ryzen с 3D V-Cache.
- **Проверка корректности** – стресс-тесты аллокации, чтения данных и выживаемости объектов после сборки.

## 🧱 Архитектура аллокатора

Аллокатор организован по трёхуровневой схеме:

1. **MSpan** – регион памяти (несколько страниц), разбитый на блоки одного класса размера. Содержит битовые маски для отслеживания занятых (`alloc_bits`) и помеченных GC (`marked_bits`) объектов. Максимум 256 блоков на спан.
2. **MCache** – локальный для каждого потока кэш, который без блокировок выделяет объекты из текущего спана. При исчерпании свободного места запрашивает новый спан из `MCentral`. Порядок операций — `find_next_block → write_to_slot → publish_block` — гарантирует, что бит в `alloc_bits` выставляется строго после записи данных (важно для корректности конкурентного GC).
3. **MCentral** – глобальный пул спанов с sharded lock: каждый класс размера имеет отдельный `Mutex`, чтобы потоки не блокировали друг друга при работе с разными классами. Хранит раздельные списки `partial` (есть свободные слоты) и `full` (заполненные). После GC-цикла `sweep_full_spans` перекладывает освободившиеся спаны из `full` обратно в `partial`.
4. **ManagedHeap** – отвечает за виртуальное адресное пространство: выделяет страницы под новые спаны, хранит карту страниц (`page_map`) и предоставляет низкоуровневый доступ на чтение/запись по `ObjectId`.

Сборщик мусора (`GarbageCollector`) реализует классический Mark & Sweep:
- `initialize` – начальная разметка корневых объектов, вызов `start_marking()`.
- `trace` – обход достижимых объектов и пометка в `marked_bits`.
- `sweep` – освобождение всех непомеченных блоков, сброс масок, вызов `sweep_full_spans` в `MCentral` для возврата спанов в оборот, вызов `stop_marking()`.

## 📊 Результаты производительности

Бенчмарки проведены на двух платформах, сборка `--release`. Тест 4/5 использует смешанное распределение: 60% мелких (64 байта), 30% средних (512 байт), 10% крупных (3072 байта), 8 потоков, 4 млн объектов суммарно.

**Тестовые стенды:**
- **macOS** — Apple M-серия, 11 физических ядер, 18 GB RAM
- **CachyOS** — AMD Ryzen 7 5700X3D (96 MB 3D V-Cache), Linux

## 🔬 Интерпретация результатов

**Наш аллокатор — специализируется на многопоточности.** Его архитектура (per-thread MCache + sharded MCentral + preалlocated heap) полностью устраняет contention между потоками за счёт изоляции горячего пути аллокации. Это напрямую реализует идеи Go runtime и TCMalloc.

**Влияние платформы критично.** Одна и та же кодовая база показывает принципиально разные относительные результаты:
- Против ptmalloc2 на Linux: 4.6–5.1x в многопоточности — ptmalloc2 плохо масштабируется под конкуренцией.
- Против jemalloc на macOS: паритет — одинаковая архитектура.
- Против jemalloc на Ryzen 5700X3D: 1.3x в нашу пользу — большой L3 усиливает cache-friendly поведение sharded-lock.
- Против mimalloc: проигрываем в один поток везде (~1.7–3.4x), выигрываем в многопоточности на Linux (~1.9x) и ~паритет на macOS.

**В однопоточном режиме** мы всегда либо на паритете с jemalloc, либо проигрываем более современным аллокаторам (mimalloc, libmalloc). Это неизбежный trade-off: атомарные операции в `find_next_block` и `publish_block` стоят дороже, чем bump-pointer в mimalloc, и эти издержки не окупаются без конкуренции за память.

**Стабильность** — отличительная черта нашего аллокатора: разброс между прогонами минимален (в отличие от mimalloc на Linux и jemalloc для крупных объектов). Это следствие отсутствия системных вызовов (`madvise`, `munmap`) в горячем пути и предсказуемой работы с преаллоцированной памятью.

## 🔧 Запуск

```bash
cargo run --release
```

Запускает все стресс-тесты и бенчмарки последовательно:
- **Тест 1–3:** корректность аллокации, исчерпание спанов, Mark & Sweep GC.
- **Тест 4–5:** многопоточный бенчмарк нашего аллокатора vs системного.
- **Бенч A:** fragmentation recovery после GC-цикла.
- **Бенч B:** масштабирование GC-паузы.
- **Бенч C:** однопоточный baseline.
- **Бенч D:** разбивка производительности по классам размеров.

Для сравнения с jemalloc или mimalloc добавьте соответствующий crate в `Cargo.toml` и зарегистрируйте его как глобальный аллокатор через `#[global_allocator]`.

## 🤝 Направления развития

- Конкурентный GC: убрать STW вокруг фазы `trace`, задействовав уже заложенную инфраструктуру write barrier (`write_reference`, `barrier_worklist`, `is_marking`).
- Расширение таблицы size-классов (промежуточные классы между 512 и 3072 байтами) для снижения внутренней фрагментации.
- Поддержка крупных объектов (large-object spans, >3072 байт) с прямым выделением страниц в обход spin-классов.
- Возврат пустых страниц ОС (аналог `madvise`/`munmap`) для ограничения RSS при долгой работе.
- Перенос бенчмарков в `criterion` для получения статистически значимых результатов с доверительными интервалами вместо единичных замеров.
