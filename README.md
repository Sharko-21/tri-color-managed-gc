# managed_gc – Управляемый рантайм с мультипоточной аллокацией и сборщиком мусора

**managed_gc** — это экспериментальная реализация управляемой кучи, многопоточного lock-free аллокатора и сборщика мусора (Mark & Sweep) на языке Rust.  
Проект вдохновлён архитектурой аллокатора Go: спаны памяти, локальные кэши потоков и центральный пул. Он предназначен для исследовательских целей и демонстрации возможностей языка при построении высокопроизводительных управляемых сред.

## 🚀 Ключевые возможности

- **Мультипоточная аллокация без глобальных блокировок** – каждый поток использует локальный кэш (`MCache`) и атомарные операции, что минимизирует конкуренцию.
- **Собственная система классов размеров** – от 64 байт до 3072 байт, с автоматическим подбором спанов нужной ёмкости.
- **Сборщик мусора (Mark & Sweep)** – корректно обрабатывает циклические ссылки и освобождает недостижимые объекты.
- **Переиспользование памяти после GC** – освобождённые спаны возвращаются в центральный пул (`MCentral`) через `sweep_full_spans` и доступны для повторной аллокации без обращения к ОС.
- **Write barrier (инфраструктура)** – заложена инфраструктура для инкрементального GC: `write_reference`, `barrier_worklist`, флаг `is_marking`.
- **Профилирование производительности** – сравнение с macOS libmalloc, jemalloc и mimalloc в многопоточном и однопоточном режимах.
- **Проверка корректности** – стресс-тесты аллокации, чтения данных и выживаемости объектов после сборки.

## 🧱 Архитектура аллокатора

Аллокатор организован по трёхуровневой схеме:

1. **MSpan** – регион памяти (несколько страниц), разбитый на блоки одного класса размера. Содержит битовые маски для отслеживания занятых (`alloc_bits`) и помеченных GC (`marked_bits`) объектов. Максимум 256 блоков на спан.
2. **MCache** – локальный для каждого потока кэш, который без блокировок выделяет объекты из текущего спана. При исчерпании свободного места запрашивает новый спан из `MCentral`. Порядок операций — `find_next_block → write_to_slot → publish_block` — гарантирует, что бит в `alloc_bits` выставляется строго после записи данных (важно для корректности конкурентного GC).
3. **MCentral** – глобальный пул спанов с sharded lock: каждый класс размера имеет отдельный `Mutex`, чтобы потоки не блокировали друг друга при работе с разными классами. Хранит раздельные списки `partial` (есть свободные слоты) и `full` (заполненные). После GC-цикла `sweep_full_spans` перекладывает освободившиеся спаны из `full` обратно в `partial`.
4. **ManagedHeap** – отвечает за виртуальное адресное пространство: выделяет страницы под новые спаны, хранит карту страниц (`page_map`) и предоставляет низкоуровневый доступ на чтение/запись по `ObjectId`.

Сборщик мусора (`GarbageCollector`) реализует классический Mark & Sweep:
- `initialize` – начальная разметка корневых объектов, вызов `start_marking()`.
- `trace` – обход достижимых объектов и пометка в `marked_bits`.
- `sweep` – освобождение всех непомеченных блоков, сброс масок, вызов `sweep_full_spans` в `MCentral` для возврата спанов в оборот, вызов `stop_marking()`.

## 📊 Результаты производительности

Все измерения проведены на Apple M-серии (11 физических ядер, 18 GB RAM), сборка `--release`. Тест 4/5 использует смешанное распределение объектов: 60% мелких (64 байта), 30% средних (512 байт), 10% крупных (3072 байта).

### Тест 4/5 — многопоточная аллокация (8 потоков, 4 млн объектов)

Сравнение нашего аллокатора с тремя разными системными аллокаторами.

#### vs macOS libmalloc (стандартный аллокатор Rust на macOS)

| Запуск | Наш аллокатор | libmalloc | Результат |
|--------|---------------|-----------|-----------|
| 1 | 315 ms | 419 ms | 🎉 быстрее в 1.33x |
| 2 | 311 ms | 490 ms | 🎉 быстрее в 1.57x |
| 3 | 298 ms | 423 ms | 🎉 быстрее в 1.42x |

Наш аллокатор стабильно выигрывает у libmalloc под многопоточной нагрузкой за счёт sharded lock в `MCentral` (каждый size-класс — отдельный мьютекс) и отсутствия глобального лока на пути аллокации.

#### vs jemalloc

| Запуск | Наш аллокатор | jemalloc | Результат |
|--------|---------------|----------|-----------|
| 1 | 326 ms | 270 ms | 📈 медленнее в 1.21x |
| 2 | 296 ms | 277 ms | 📈 медленнее в 1.07x |
| 3 | 253 ms | 294 ms | 🎉 быстрее в 1.16x |
| 4 | 269 ms | 267 ms | ~паритет (1.01x) |

С jemalloc — устойчивый паритет. Это закономерно: jemalloc реализует те же архитектурные идеи (per-thread cache + size classes + sharded central pool), что и наш аллокатор. Совпадение результатов подтверждает правильность архитектурных решений.

#### vs mimalloc

| Запуск | Наш аллокатор | mimalloc | Результат |
|--------|---------------|----------|-----------|
| 1 | 266 ms | 346 ms | 🎉 быстрее в 1.30x |
| 2 | 243 ms | 479 ms | 🎉 быстрее в 1.97x |
| 3 | 251 ms | 254 ms | ~паритет (1.01x) |
| 4 | 247 ms | 227 ms | 📈 медленнее в 1.09x |

mimalloc показывает наибольший разброс — от паритета до почти двукратного проигрыша нашему аллокатору. Это связано с тем, что mimalloc оптимизирован для минимальных накладных расходов на одну операцию (bump-pointer внутри сегментов), но менее предсказуем под высокой конкуренцией потоков.

### Бенч C — однопоточный baseline

Тот же паттерн аллокации (60/30/10%), один поток, 2 млн объектов.

| Аллокатор | Наш | Системный | Соотношение |
|-----------|-----|-----------|-------------|
| libmalloc | ~130 ms | ~85 ms | проигрываем в 1.67–1.98x |
| jemalloc | ~125 ms | ~105 ms | ~паритет (1.04–1.27x) |
| mimalloc | ~130 ms | ~55 ms | проигрываем в **1.78–3.40x** |

В однопоточном режиме наш аллокатор проигрывает всем трём — особенно mimalloc, который является одним из самых быстрых однопоточных аллокаторов в индустрии. Весь выигрыш нашего аллокатора сосредоточен именно в многопоточных сценариях за счёт устранения contention.

---

### Бенч D — разбивка по классам размеров (один поток, 1 млн объектов)

Изолированное измерение каждого size-класса позволяет выявить, где именно находится отставание.

| Класс | Наш | libmalloc | jemalloc | mimalloc |
|-------|-----|-----------|----------|----------|
| B64 (64 байта) | ~20–27 ms | ~13–15 ms | ~13 ms | ~7–8 ms |
| B512 (512 байт) | ~54–78 ms | ~15–48 ms* | ~42–48 ms | ~15–22 ms |
| B3072 (3072 байта) | ~303–363 ms | ~307–340 ms | ~307–340 ms | ~254–339 ms |

*Разброс B512 vs libmalloc связан с методологическим нюансом: каждая аллокация включает создание временного `Vec<u8>` для payload через системный аллокатор, что искажает результат для мелких классов.

Ключевые наблюдения:
- **B3072** — наш аллокатор на уровне или лучше libmalloc и jemalloc, незначительно хуже mimalloc. Для крупных объектов стоимость копирования данных доминирует над накладными расходами на управление спанами.
- **B64** — проигрываем ~1.5–1.8x. При 64 блоках на спан частота обращений к `MCentral` невысока, но сама операция включает несколько атомарных load/store, тогда как быстрые аллокаторы используют почти бесплатный bump-pointer.
- **B512** — самый сложный класс: всего 8 блоков на спан при стандартной конфигурации означает высокую частоту пополнения спанов, что создаёт нагрузку на `MCentral` даже в один поток.

---

### Бенч A — Fragmentation Recovery

Проверяет, реально ли освобождённая память возвращается в оборот после GC, а не продолжает занимать страницы.

**Сценарий:** создать 20 000 объектов B64, половину сделать мусором, запустить GC, затем создать ещё 20 000 объектов.

| Метрика | Значение |
|---------|----------|
| Занятых страниц после волны 1 | 469 |
| Занятых страниц после волны 2 | 469 |
| Прирост страниц | **0** |

Результат стабилен во всех прогонах: вторая волна аллокаций полностью умещается в страницы, освобождённые GC, без выделения новых страниц у ОС. `sweep_full_spans` в `MCentral` корректно перекладывает освободившиеся спаны из `full` в `partial`.

---

### Бенч B — масштабирование GC-паузы

Измерение времени каждой фазы GC (`initialize`, `trace`, `sweep`) в зависимости от размера живого множества.

| Живых объектов | initialize | trace | sweep | Итого |
|----------------|------------|-------|-------|-------|
| 10 000 | ~115 µs | ~7 µs | ~20 µs | ~143 µs |
| 100 000 | ~1.2 ms | ~110 µs | ~35 µs | ~1.35 ms |
| 500 000 | ~6.0 ms | ~600 µs | ~170 µs | ~6.8 ms |

Время растёт примерно линейно с количеством объектов — ожидаемое поведение для трассирующего Mark & Sweep без патологических скачков. `initialize` доминирует: он линейно проходит по корням и первым обращается к каждой странице кучи (эффект page fault на холодной памяти). `trace` заметно быстрее, так как работает только с достижимыми объектами через их собственные ссылки.

## 🔬 Интерпретация результатов

Данные позволяют сформулировать точный тезис о том, где именно работает наш аллокатор:

**Против libmalloc:** выигрываем ~1.4–1.6x в многопоточности, проигрываем ~1.7–2.0x в однопоточности. libmalloc нативно оптимизирован под Apple Silicon, но имеет contention под нагрузкой.

**Против jemalloc:** паритет — архитектурные идеи совпадают (per-thread cache + size classes + sharded pool).

**Против mimalloc:** проигрываем в однопоточности (mimalloc — один из быстрейших в индустрии для этого сценария), в многопоточности результаты нестабильны, но иногда выигрываем.

**Вывод:** специализированный size-class аллокатор с sharded lock имеет смысл именно в сценарии высокой многопоточной нагрузки с однородным распределением размеров объектов. В однопоточном коде он проигрывает современным общецелевым аллокаторам из-за накладных расходов на атомарные операции и управление спанами, которые не компенсируются устранением contention.

## 🔧 Запуск

```bash
cargo run --release
```

Запускает все стресс-тесты и бенчмарки последовательно:
- **Тест 1–3:** корректность аллокации, исчерпание спанов, Mark & Sweep GC.
- **Тест 4–5:** многопоточный бенчмарк нашего аллокатора vs системного.
- **Бенч A:** fragmentation recovery после GC-цикла.
- **Бенч B:** масштабирование GC-паузы.
- **Бенч C:** однопоточный baseline.
- **Бенч D:** разбивка производительности по классам размеров.

Для сравнения с jemalloc или mimalloc добавьте соответствующий crate в `Cargo.toml` и зарегистрируйте его как глобальный аллокатор через `#[global_allocator]`.

## 🤝 Направления развития

- Конкурентный GC: убрать STW вокруг фазы `trace`, задействовав уже заложенную инфраструктуру write barrier (`write_reference`, `barrier_worklist`, `is_marking`).
- Расширение таблицы size-классов (промежуточные классы между 512 и 3072 байтами) для снижения внутренней фрагментации.
- Поддержка крупных объектов (large-object spans, >3072 байт) с прямым выделением страниц в обход spin-классов.
- Возврат пустых страниц ОС (аналог `madvise`/`munmap`) для ограничения RSS при долгой работе.
- Перенос бенчмарков в `criterion` для получения статистически значимых результатов с доверительными интервалами вместо единичных замеров.
