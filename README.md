# Managed GC (Rust Precise Garbage Collector)

Экспериментальный рантайм управления памятью на языке Rust. Проект реализует сборщик мусора на базе **трёхцветной маркировки (Three-Color Mark-Sweep)**, работающий поверх монолитной типизированной кучи с ручной сериализацией объектов и выравниванием памяти.

## 🚀 Текущий статус архитектуры

Проект находится в фазе стабильного MVP неперемещающего сборщика мусора (Non-compacting GC). На текущем этапе полностью реализованы и покрыты интеграционными тестами следующие компоненты:

1. **Топология кучи (`ManagedHeap` & `Allocator`)**:
* Память выделяется единым сырым буфером байтов (`Vec<u8>`).
* Реализован быстрый `BumpAllocator`, выделяющий память за $O(1)$.
* Автоматическое выравнивание границ полезной нагрузки (Payload Alignment) до **8 байт** для исключения невыровненного чтения на уровне CPU.


2. **Низкоуровневая сериализация (`ObjectHeader`)**:
* Каждый объект снабжен фиксированным мета-заголовком размером **40 байт**.
* Заголовок атомарен, знает свой физический размер (`size`), размер payload, количество исходящих ссылок (`refs_count`) и текущий маркер GC (`GcColor`).


3. **Алгоритм сборщика мусора (`GarbageCollector`)**:
* **Итеративный обход (`Mark`)**: Фаза трассировки графа разворачивается через явный стек (`worklist` в куче) без использования рекурсии, что гарантирует защиту от `Stack Overflow` при любой глубине вложенности объектов.
* **Линейная очистка (`Sweep`)**: Фаза сканирования совершает последовательный проход по всей аллоцированной памяти от смещения `0` до текущего указателя аллокатора. Безопасность шага обеспечивается монолитным сохранением геометрии объектов (`header.size`) в заголовках, в том числе мёртвых.

## 📐 Спецификация раскладки памяти (Memory Layout)

Каждая аллокация в куче формирует непрерывный бинарный блок следующей структуры:

```text
┌───────────────────────────────────────┬───────────────────────────┬───────────────────────────┐
│           ObjectHeader                │    Array of ObjectIds     │     Aligned Payload       │
│            (40 байт)                  │   (refs_count * 8 байт)   │     (align_up to 8B)      │
└───────────────────────────────────────┴───────────────────────────┴───────────────────────────┘

```

* **Инвариант целостности**: Даже если объект признан мусором, его заголовок и поле `size` сохраняются. Это позволяет фазе `sweep` использовать мёртвые объекты как «дорожные указатели» для прыжка к следующему валидному адресу.

## 🎨 Трехцветный инвариант (GC Invariant)

Сборщик мусора оперирует тремя состояниями объекта:

* **`White` (Белый)**: Объект не посещен. В начале цикла — это все объекты. В конце фазы `trace` — это недостижимый мусор.
* **`Grey` (Серый)**: Объект посещен, добавлен в `worklist`, но его дочерние связи еще не исследованы.
* **`Black` (Черный)**: Объект посещен, все его прямые зависимости исследованы и перекрашены в серый.

> **Строгое правило**: Черный объект никогда не указывает на белый. Если в процессе линейного сканирования `sweep` обнаруживается серый объект — рантайм немедленно генерирует критическую панику, сигнализирующую о нарушении консистентности графа.

## 📦 Структура модулей

* `src/datastruct.rs` — Определение базовых примитивов (`ObjectId`, `GcColor`, `ObjectHeader`) и структуры пользовательского представления объекта `ManagedObject`.
* `src/allocator.rs` — Трейт и абстракции низкоуровневого управления памятью.
* `src/bump_allocator.rs` — Реализация линейного аллокатора со смещением указателя.
* `src/managed_heap.rs` — Высокоуровневый интерфейс кучи, инкапсулирующий чтение/запись сырых байт и проксирование заголовков.
* `src/garbage_collector.rs` — Оркестратор фаз сборки мусора (`initialize` $\rightarrow$ `trace` $\rightarrow$ `sweep`).

## 🧪 Верификация и тестирование

Проект покрыт интеграционными тестами, проверяющими граничные состояния рантайма.

### Сценарии в `mod tests`:

1. **`test_heap_serialization_and_alignment`**: Проверка маршаллинга объектов, корректности расчета паддинга (Padding) для невыровненных payload и детекция ошибки `OutOfMemory`.
2. **`test_garbage_collector_lifecycle`**: Симуляция сложного графа владения с **циклической ссылкой** (два изолированных объекта ссылаются друг на друга, но отрезаны от корней). Тест верифицирует, что:
* Живой корень успешно перекрашивается и выживает.
* Циклический мусор изолируется, а его внутренние связи (`refs_count`) сбрасываются в `0`, предотвращая утечку памяти.



### Запуск тестов:

```bash
cargo test

```

### Запуск бенчмарков:

```bash
cargo bench

```

## 🗺️ Вектор дальнейшего развития (Roadmap)

Текущая неперемещающая Stop-The-World модель на базе Bump-аллокатора накладывает ограничения на повторное использование памяти (указатель аллокатора движется только вперед). На очереди стоит масштабный рефакторинг подсистемы выделения памяти:

* **Переход на многоуровневый Free-List аллокатор**: Реализация концепции, вдохновленной TCMalloc (Go Runtime):
* `MCache` — потоковые локальные кэши со списками свободных блоков, разбитых на фиксированные классы размеров (Size Classes: 16B, 32B, 64B, 128B).
* `MCentral` — централизованные пулы страниц памяти (`MSpan`).
* `MHeap` — глобальный менеджер страниц кучи.


* **Модификация GC под Free-List**: Перевод фазы `sweep` с простого зануления линков на физический возврат освободившихся блоков памяти в структуру `FreeList` для их повторного переиспользования за $O(1)$ без необходимости дефрагментации.

* **Concurrent GC**: Добавление функционала параллельной сборки мусора, чтобы максимально снизить STW фазу
