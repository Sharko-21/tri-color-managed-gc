# managed_gc – Управляемый рантайм с мультипоточной аллокацией и сборщиком мусора

**managed_gc** — это экспериментальная реализация управляемой кучи, многопоточного lock-free аллокатора и сборщика мусора (Mark & Sweep) на языке Rust.  
Проект вдохновлён архитектурой аллокатора Go: спаны памяти, локальные кэши потоков и центральный пул. Он предназначен для исследовательских целей и демонстрации возможностей языка при построении высокопроизводительных управляемых сред.

## 🚀 Ключевые возможности

- **Мультипоточная аллокация без глобальных блокировок** – каждый поток использует локальный кэш (`MCache`) и атомарные операции, что минимизирует конкуренцию.
- **Собственная система классов размеров** – от 64 байт до 3072 байт, с автоматическим подбором спанов нужной ёмкости.
- **Сборщик мусора (Mark & Sweep)** – корректно обрабатывает циклические ссылки и освобождает недостижимые объекты.
- **Профилирование производительности** – сравнение со стандартным аллокатором Rust в многопоточном режиме с реалистичным распределением размеров объектов.
- **Проверка корректности** – стресс-тесты аллокации, чтения данных и выживаемости объектов после сборки.


- **`src/main.rs`** – стресс-тесты и сравнительные бенчмарки аллокации.
- **`src/datastruct.rs`** – базовые типы: `ManagedObject`, `ObjectId`, `TypeDescriptor`.
- **`src/garbage_collector.rs`** – реализация Mark & Sweep GC.
- **`src/managed_heap.rs`** – управляемая куча с виртуальной адресацией и чтением/записью по `ObjectId`.
- **`src/mcache.rs`** – локальный кэш аллокации для потоков (lock-free).
- **`src/mcentral.rs`** – центральный пул спанов.
- **`src/mspan.rs`** – спан памяти с битовыми масками занятости и пометок для GC.

## 🧱 Архитектура аллокатора

Аллокатор организован по трёхуровневой схеме:

1. **MSpan** – регион памяти (несколько страниц), разбитый на блоки одного класса размера. Содержит битовые маски для отслеживания занятых и помеченных объектов. Максимум 256 блоков на спан.
2. **MCache** – локальный для каждого потока кэш, который без блокировок выделяет объекты из текущего спана. При исчерпании свободного места запрашивает новый спан из `MCentral`.
3. **MCentral** – глобальный пул спанов. Предоставляет новые спаны по запросу кэшей. `ManagedHeap` отвечает за виртуальное адресное пространство и размещение данных по фиксированным идентификаторам `ObjectId`.

Сборщик мусора (`GarbageCollector`) реализует классический Mark & Sweep:
- `initialize` – начальная разметка корневых объектов.
- `trace` – обход достижимых объектов и пометка битовой маски.
- `sweep` – освобождение всех непомеченных блоков и сброс масок для следующего цикла.

## 📊 Результаты производительности

### Лёгкая нагрузка (8 потоков, 400 тыс. объектов)

Ниже приведены примеры многократных запусков стресс-теста, сравнивающих наш аллокатор с системным (`std::alloc`) при выделении 400 000 объектов в 8 потоках (с распределением 60% мелких, 30% средних, 10% крупных).

| Запуск | Наш аллокатор | Системный аллокатор | Результат               |
|--------|---------------|---------------------|--------------------------|
| 1      | 42.18 ms      | 63.26 ms            | 🎉 быстрее в 1.50 раз   |
| 2      | 35.39 ms      | 71.45 ms            | 🎉 быстрее в 2.02 раз   |
| 3      | 33.33 ms      | 39.50 ms            | 🎉 быстрее в 1.19 раз   |
| 4      | 35.86 ms      | 28.23 ms            | 📈 медленнее в 1.27 раз |
| 5      | 32.21 ms      | 25.51 ms            | 📈 медленнее в 1.26 раз |
| 6      | 37.73 ms      | 63.67 ms            | 🎉 быстрее в 1.69 раз   |

Наш lock-free аллокатор часто опережает системный, особенно на смешанных нагрузках, но иногда уступает (особенно после прогрева) — типичное поведение для lock-free структур, зависящее от стечения операций потоков

### Высокая нагрузка (11 потоков, 4,4 млн объектов)

При увеличении числа потоков до 11 и объёма аллокаций до 4,4 млн объектов разница становится **устойчивой и однозначной**:

| Запуск | Наш аллокатор | Системный аллокатор | Результат               |
|--------|---------------|---------------------|--------------------------|
| 1      | 451.16 ms     | 650.62 ms           | 🎉 быстрее в 1.44 раз   |
| 2      | 436.49 ms     | 672.58 ms           | 🎉 быстрее в 1.54 раз   |
| 3      | 420.73 ms     | 657.06 ms           | 🎉 быстрее в 1.56 раз   |
| 4      | 427.90 ms     | 681.23 ms           | 🎉 быстрее в 1.59 раз   |
| 5      | 431.86 ms     | 666.74 ms           | 🎉 быстрее в 1.54 раз   |

**Среднее ускорение:** ~1.53x. Наш lock-free аллокатор стабильно обгоняет системный на 44–59% при высокой многопоточной нагрузке. Это достигается за счёт:

- отсутствия глобальных блокировок на пути аллокации;
- эффективного кэширования спанов в каждом потоке;
- атомарных битовых масок вместо сложных структур данных.

## 🔧 Запуск

### Стресс-тесты и сравнение аллокаторов

```bash
cargo run --release
```

Бенчмарки замеряют вызов `gc.collect()` на чистых кучах с разным количеством объектов, в том числе с циклическим мусором и сотнями мёртвых объектов, чтобы оценить скорость sweep-фазы.
